# sea.AI Badcase 排查

## 核心原则

先确认是否使用本 Skill；确认后，先建立业务期望和应有接待链路，再以 Trace 为执行证据、以 sea.AI 当前配置为定位依据，输出可直接执行的配置优化建议。

当前 Trace 命中的意图或 Agent 只是“实际接待方”，不是“正确接待方”的默认前提。必须先判断问题应由哪个意图或 Agent 接待，再决定根因位于应用层路由边界，还是当前 Agent 内部的专家、汇总、转人工或质检配置。

sea.AI 配置检查严格只读。不得编辑字段、保存草稿、发布版本或触发调试。

## 工作流

1. 识别用户是否在请求 sea.AI 机器人调试结果或 Badcase 分析。
2. 先询问：`是否使用 sea.AI Badcase 排查 Skill 进行分析？`
3. 未得到确认前，不打开调试链接，不分析 Trace，不进入配置页。
4. 用户确认后，读取固定说明文档：
   `https://wdh.feishu.cn/wiki/QQkjwYQSkiAJUJkHF8FciXOCn5d?from=from_copylink`
5. 在下钻 Trace 前，先确认或主动提出本类问题的业务期望：机器人直接回答、转人工，还是先回答后转人工；同时确认期望由哪个意图或 Agent 接待。用户未明确时，根据问题类型、业务说明和应用中可见的接待规则作初步判断，并把推断写明，不得悄悄把当前命中结果当作业务期望。
6. 打开用户提供的调试记录，先识别应用 UID、应用版本和实际命中意图，不把该命中结果当作正确路由结论。
7. 只读进入 Trace 对应的 sea.AI 应用，核对应用层全部相关 Agent 接待规则及接待顺序，识别当前接待方和可能的正确接待方。重点检查特殊场景、答疑意图等规则之间是否存在范围重叠、优先级截流或兜底误接待。
8. 回到 Trace 按文档说明核对完整执行链路，建立“期望接待链路 vs 实际接待链路”对照，并提取会话 ID、用户原话、最终回复、耗时、意图、变量、专家、FAQ、工具、转人工判断、最终输出和质检结果。
9. 逐层核对 Trace 涉及及业务期望涉及的 sea.AI 当前配置：
   - 应用层：应用设置、Agent 接待规则、接待顺序、意图描述、接待 Agent 与版本；同时核对当前意图与候选正确意图的边界。
   - 子 Agent 层：任务执行规则、回复输出规则、转人工场景、知识库、变量、质检方案。
   - 专家层：专家接待规则、回复标准、工具及专家配置。
10. 先判断路由是否合理，再分析当前 Agent 内部执行。将技术节点映射成可操作的配置位置；节点名只作为 Trace 证据，不用节点名代替配置动作。
11. 完整读取 [report-template.md](references/report-template.md)，逐项填充并按原结构输出。

## 只读边界

- 可以打开页面、展开配置、切换只读标签和读取提示词。
- 不得修改输入框、开关、下拉框或版本选择。
- 不得点击保存、发布、创建、删除、发送、预览调试。
- 每次都重新读取当前配置；不得用历史报告中的提示词冒充现有配置。
- 无法访问配置时，标记为“配置未核实”，不得猜测位置、原提示词或规则。

## 业务期望与路由边界判断

每次分析都必须先回答以下问题：

1. 这类问题的期望处理方式是什么：直接回答、转人工，还是先回答后转人工？
2. 期望接待链路是什么？至少写到“应用层意图 / Agent → 子 Agent 或专家 / FAQ / 知识库 → 最终处理”。
3. 实际接待链路是什么？以 Trace 和当前配置共同证明。
4. 当前命中的意图或 Agent 是否应接待本问题？如果不应，哪个意图或 Agent 更符合业务职责？
5. 首次偏差发生在应用层接待边界，还是正确路由后的下游执行？

判断规则：

- 不得因为 Trace 命中了某个意图或 Agent，就默认它是正确接待方。
- 如果当前命中的是以强制转人工为职责的特殊场景 Agent，先判断用户问题是否真的属于强制转人工场景。
- 对本应由答疑意图、FAQ 或知识库直接回答的问题，即使实际 Trace 没有知识库调用、没有专家生成答疑结果，也要把“被特殊场景提前截流”作为路由边界候选根因进行核查。
- 如果特殊场景 Agent 的固定输出策略就是转人工，那么其内部知识库或专家是否生成正确答案，不一定决定用户最终输出；不得把“专家是否答对”当作主根因。
- 若问题不应由当前 Agent 接待，主根因优先定位到应用层 `Agent 接待规则` 的意图描述、边界重叠或接待顺序；当前 Agent 内的固定转人工、汇总覆盖等通常是该错误路由下的预期结果或放大环节。
- 若确认当前 Agent 本就应接待，再继续定位子 Agent、专家、汇总、转人工和质检配置。

示例：

> 业务期望：普通资金问答由「筹-答疑意图」接待，经 FAQ / 知识库直接回答。
>
> 实际链路：问题被「筹-特殊场景」提前接待，进入固定转人工链路。
>
> 边界结论：当前 Agent 不应接待；根因优先定位到应用层 `Agent 接待规则` 中「筹-特殊场景」边界过宽，并同步检查「筹-答疑意图」承接规则是否不足。

## Trace 与配置映射

| Trace 节点 | 报告中的配置表达 |
|---|---|
| `process_chat_core` | 应用调用入口、应用版本、用户输入和历史消息 |
| `intention_recognition_node` | 应用层 → Agent 接待规则 → 对应意图 |
| `business_process_start_node` | 应用层最终意图及接待 Agent |
| `planning_concurrency_node` | 子 Agent 层 → 任务专家配置 → 对应专家 → 专家接待规则 |
| `faq_node`、`reason` | 子 Agent 知识库及专家回复标准 |
| `tool_node` | 专家层 → 对应专家 → 工具 |
| `post_processing` | 子 Agent 层 → 任务执行规则 → 转人工场景 |
| `aggregate_expert_results_node` | 子 Agent 层 → 任务执行规则 → 回复输出规则 |
| `plan_quality_check` | 子 Agent 层 → 规则 → 质检方案 → 质检规则 |
| `post_process_end_node` | 子 Agent 层 → 任务执行规则 → 最终转人工处理 |

若平台版本的配置名称发生变化，以页面当前可见名称为准，并保留 Trace 节点作证据。

`intention_recognition_node` 只证明实际识别结果。判断它是否合理时，必须同时查看应用层其他相关 Agent 接待规则和接待顺序，不能只查看被命中的那一项。

## 分析要求

- 结论前置，明确“正常 / 异常 / 证据不足”。
- 结论必须先给出业务期望、期望接待链路、实际接待链路，以及当前意图 / Agent 是否应接待。
- 根因定位先判断应用层路由边界，再下钻当前 Agent 内部执行；不得默认实际路由正确。
- 区分首次异常与后续放大环节，不把所有经过节点都列为根因。
- 判断知识库是否命中、专家是否采用、工具是否调用，三者不得混为一谈。
- 如果发生二次意图识别，以最新一组相关节点为最终结果。
- 对“未知”变量遵守平台语义：本轮提取为未知不等于覆盖已有变量。
- 优化动作固定按 P0、P1、P2 排序；不存在某级问题时仍保留该优先级小标题，并填写“本次无对应优化动作”。存在错误路由时，P0 优先收窄错误接待方的边界并补强正确意图 / Agent 的承接规则，P1 再处理当前或正确接待方的内部执行，P2 处理下游兜底与质检。
- 每个优化动作必须包含根因、配置对比表和优化目的。
- 原提示词/规则必须来自本次 sea.AI 页面；建议文本可以重写，但不得改变业务事实或凭空发明政策。

## 输出约束

- [report-template.md](references/report-template.md) 是强制输出契约，不是参考示例。
- 固定输出且只输出以下五个章节，顺序和名称不得改变：
  1. `基本信息`
  2. `排查结论`
  3. `关键链路与配置定位`
  4. `根因与优化策略`
  5. `建议复测用例与验收标准`
- 不得增加“详细执行过程”“修复优先级”“总结”等独立章节，不得拆分或合并上述五章。
- 数据缺失时保留对应字段并填写“无”；sea.AI 配置无法读取时填写“配置未核实”。
- 保持精炼，不重复叙述表格中已经表达的信息。
- 基本信息只列 Trace ID、会话 ID、总耗时和调试链接。
- 用户原话、实际最终回复、专家节点回复、预期结果使用四行两列表格。
- 排查结论必须包含“业务期望与路由边界判断”表，列明期望处理方式、期望接待链路、实际接待链路、当前意图 / Agent 是否应接待及边界结论。
- 关键链路表保留配置位置与 Trace 证据。
- 每个配置优化对比表固定为：

| 项目 | 原提示词/规则 | 建议更新后提示词/规则 |
|---|---|---|
| 配置位置 | 当前页面的完整层级路径 |  |
| 优化对比 | 当前页面原文 | 建议的新文本 |

- 配置位置单独占一行；原规则与新规则只在“优化对比”行横向比较。
- 复测用例和验收标准合并在最后一个章节。

## 常见错误

| 错误 | 正确做法 |
|---|---|
| 只写 `aggregate_expert_results_node` | 写出子 Agent 层的具体配置路径，再附节点证据 |
| 没看 sea.AI 就给原提示词 | 先只读核对当前配置；无法访问则标记未核实 |
| 把报告写成逐节点流水账 | 只保留关键链路、根因和可执行优化动作 |
| 默认当前命中的 Agent 就是正确接待方 | 先比较期望接待链路与实际接待链路，再判断是否需要下钻当前 Agent |
| Trace 没有 FAQ / 专家答疑结果就排除路由问题 | 即使下游未执行，也要核查问题是否被特殊场景或高优先级意图提前截流 |
| 特殊场景固定转人工却把汇总节点当主根因 | 先判断特殊场景是否应接待；不应接待时优先修复应用层意图边界和接待顺序 |
| 把固定话术质检通过解释为业务正确 | 区分格式安全与业务相关性 |
| 给出模糊建议“优化提示词” | 给出位置、原规则和可直接替换的新规则 |
